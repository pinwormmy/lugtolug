# Agent Instructions

- "배포해" 또는 "배포해줘"는 현재 worktree 변경사항을 검증한 뒤 `main`에 직접 push해서 자동 배포까지 진행하라는 뜻이다.
- 이 저장소는 `main` push 시 자동 배포되므로, 배포 지시를 받으면 PR을 만들지 않는다.
- 각 worktree의 작업은 가능한 한 작업 단위별 로컬 커밋으로 정리한다. 단, 최종 배포 반영은 배포 절차에서 최신 `origin/main` 위에 재적용한 뒤 수행한다.
- 배포 지시 실행 순서:
  1. 현재 브랜치와 변경사항을 확인한다.
  2. 현재 HEAD가 detached 상태이면 현재 커밋/변경사항을 임시 배포 브랜치로 보존한다.
  3. 현재 요청과 관련된 변경사항만 stage한다.
  4. 가능한 테스트와 빌드 검증을 실행한다.
  5. 검증이 통과하면 `origin/main`을 fetch해서 최신 상태를 확인한다.
  6. 원격 `main`이 앞서 있으면 현재 요청과 관련된 변경사항만 최신 `origin/main` 위에 재적용한다.
     - 커밋된 변경은 rebase 또는 cherry-pick을 사용한다.
     - 커밋되지 않은 변경은 stash 또는 임시 커밋으로 보존한 뒤 재적용한다.
  7. 재적용 중 충돌이 발생하면 자동 push하지 않는다. 해결 가능한 범위면 충돌을 해결하고 다시 검증하며, 불명확하면 충돌 파일과 원인을 보고한다.
  8. push 직전 `git status`와 `git log --oneline origin/main..HEAD`로 배포 범위를 확인한다.
  9. 변경사항을 커밋한다.
  10. `main`에 fast-forward 또는 병합 후 push한다.
  11. push 결과와 배포 트리거 여부를 보고한다.
- 테스트나 빌드 검증이 실패하면 push하지 말고 실패 내용을 보고한다.
- unrelated changes는 커밋하거나 되돌리지 않는다.
- PR은 사용자가 명시적으로 요청한 경우에만 생성한다.
- "PR 올려줘"는 현재 worktree 변경사항을 검증 후 브랜치에 push하고 GitHub PR을 생성하라는 뜻이다.
- "커밋만 해줘"는 검증 후 로컬 커밋까지만 수행하라는 뜻이다.
- "푸시만 해줘"는 현재 브랜치를 원격에 push하라는 뜻이다.
